# 系统架构设计文档

## 3.1 技术栈选型

| 层级 | 技术选型 | 理由 |
|------|---------|------|
| **前端** | React 18 + Ant Design 5 | 成熟、组件丰富、快速搭建后台看板 |
| **后端** | Python 3.10 + FastAPI | API开发快速、异步支持好、AI生态丰富 |
| **数据库** | PostgreSQL 15 | 结构化数据存储、支持JSON字段、稳定可靠 |
| **缓存** | Redis 7 | 缓存API调用结果、减少重复请求 |
| **定时任务** | Celery + Beat + Redis Broker | Python生态标准方案、支持周期任务 |
| **AI集成** | 智谱API SDK + 统一适配层 | 支持多模型切换 |
| **部署** | Docker + Docker Compose | 简化环境配置、便于迁移 |
| **版本控制** | Git + GitHub/GitLab | 标准方案 |

## 3.3 模块职责划分

| 模块 | 职责 | 依赖 |
|------|------|------|
| **选品算法模块** | 执行阈值筛选、计算综合评分、生成推荐清单 | Sorftime API |
| **比价监控模块** | 标题匹配、1688价格查询、变动计算、预警判定 | 1688 API、产品表 |
| **AI日报模块** | 数据聚合、Prompt构建、多模型调用、报告格式化 | 各LLM API、产品/价格表 |
| **风险规则引擎** | 加载规则库、匹配品类标签、输出风险提示 | 规则配置表 |
| **看板数据聚合** | 整合所有模块数据，提供统一的看板API | 所有内部模块 |
| **配置管理模块** | 阈值管理、模型切换、SKU关联维护 | 配置表 |

## 3.4 数据流向

### 每日数据流

```text
08:00  Sorftime API → 拉取最新品类数据 → 存入产品表（带时间戳）
         └─ 失败时：重试3次（指数退避）→ 降级使用昨日快照 → 看板提示「数据延迟」
08:30  读取产品表 → 提取关联SKU → 1688 API → 更新价格快照表
08:30  价格快照更新 → 触发风险规则引擎 → 更新产品风险标签
09:00  聚合近7天数据 → 构建Prompt → 调用GLM-5.2 → 生成日报 → 存入日报表
         └─ 主力模型失败：自动切换备用模型 → 记录 model_used
用户访问看板 → 聚合API → 实时展示
```

### 周度数据流（每周一）

```text
09:00  读取所有品类数据 → 执行选品算法 → 生成周度推荐清单 → 存入推荐表
```

## 3.5 安全方案

| 维度 | 方案 |
|------|------|
| API Key 管理 | 使用环境变量 + .env 文件（`.env` 加入 .gitignore），禁止硬编码 |
| 用户认证 | v1.0 采用访问令牌（Bearer Token），令牌以 bcrypt/argon2 哈希校验，禁止明文存储 |
| 数据加密 | 传输层 TLS 1.3；敏感字段（如配置中的密钥）不入库，必须入库的用 AES-256 加密 |
| 日志审计 | 记录所有 API 调用与关键操作；日志中对 API Key 脱敏（仅显示后4位） |
| 输入校验 | 所有用户输入经 Pydantic 校验 + 参数化查询，防 SQL 注入 |
| 防爬取 | 前端仅展示，无公开 API 暴露；管理接口需鉴权 |

## 3.6 外部 API 容错策略

所有外部 API（Sorftime / 1688 / GLM）统一通过适配层调用，具备以下能力：

| 能力 | 策略 |
|------|------|
| 重试 | 失败后指数退避重试，最多 3 次（1s / 2s / 4s）；429 限流时按 Retry-After 头等待 |
| 超时 | 单次请求 10 秒超时；AI 生成类 30 秒 |
| 断路器 | 1 分钟内失败率 >50% 触发熔断，熔断期 60 秒，期间直接走降级逻辑 |
| 降级 | Sorftime 失败 → 用昨日快照 + 看板提示「数据延迟」；GLM 失败 → 切备用模型；全部失败 → 返回缓存的上次日报 |
| 幂等 | 写操作（同步/匹配/生成）支持 Idempotency-Key，避免定时任务重复执行产生脏数据 |

## 3.7 缓存策略

| 缓存对象 | 存储 | TTL | 失效方式 |
|---------|------|-----|---------|
| Sorftime 产品列表查询结果 | Redis | 24h | 数据同步任务完成后主动清除 |
| 看板首页聚合数据 | Redis | 5min | 定时刷新 |
| 风险规则库 | 应用内存 | 启动加载 + 变更后刷新 | 配置变更广播 |
| AI 日报内容 | PostgreSQL | 永久（按日期） | — |

> 缓存 key 统一前缀 `pp:{模块}:{标识}`，便于批量清理。

## 3.8 可观测性与容量

- **三支柱**：结构化日志（JSON，含 `trace_id`）、指标（Prometheus 格式暴露 `/metrics`）、追踪（预留 OpenTelemetry 接入点）。
- **容量估算**：单用户日均约 1-2 万 SKU 拉取，月增量约 50 万行指标数据 → PostgreSQL 单机可承载，按月分区控制增长。
- **外部 API 配额**：Sorftime/1688 调用结果缓存于 Redis（TTL 24h），降低重复请求；触发限流时队列暂存。
