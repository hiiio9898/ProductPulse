# 开发规范与质量标准

## 7.1 代码规范

### Python (后端)
- 遵循 PEP 8 规范
- 使用 `black` 自动格式化，`flake8` 进行代码检查
- 行宽限制：120 字符
- 使用类型注解（Type Hints）
- Docstring 使用 Google Style

### JavaScript/React (前端)
- 使用 ESLint (Airbnb 规则) + Prettier
- 组件使用函数式组件 + Hooks
- 文件命名：组件使用 PascalCase，工具函数使用 camelCase
- 使用 PropTypes 或 TypeScript（建议 TypeScript）

### 后端目录结构

```text
backend/
├── app/
│   ├── main.py                 # FastAPI 入口
│   ├── core/                   # 配置、安全、日志
│   ├── api/v1/                 # 路由层（按模块分文件）
│   ├── models/                 # SQLAlchemy 模型
│   ├── schemas/                # Pydantic 入参/出参
│   ├── services/               # 业务逻辑（选品/比价/AI/风险）
│   ├── adapters/               # 外部 API 适配层（含重试/断路器）
│   ├── tasks/                  # Celery 定时任务
│   └── utils/
├── alembic/                    # 数据库迁移
├── tests/
└── scripts/                    # 初始化脚本
```

### 前端目录结构

```text
frontend/
├── src/
│   ├── api/                    # 接口封装（按模块）
│   ├── components/             # 通用组件
│   ├── pages/                  # 页面（首页/选品/监控/日报/配置）
│   ├── hooks/                  # 自定义 Hooks
│   ├── store/                  # 全局状态
│   └── utils/
└── public/
```

## 7.2 错误处理规范

- **后端**：统一异常处理器（FastAPI `exception_handler`），把业务异常映射为标准响应体（见 API 文档 5.1）。
- **外部 API 异常**：适配层捕获后转为本系统错误码（2001/2002），不向上抛原始栈。
- **前端**：axios 拦截器统一处理 4xx/5xx，展示 `message.error`；401 跳登录。
- **禁止吞异常**：`except` 必须记录日志或重新抛出，禁止空 `except: pass`。

```python
# 后端统一响应示例
class BizError(Exception):
    def __init__(self, code: int, message: str):
        self.code, self.message = code, message

@app.exception_handler(BizError)
async def biz_handler(_: Request, e: BizError):
    return JSONResponse({"code": e.code, "message": e.message, "data": None,
                         "timestamp": now_iso()}, status_code=200)
```

## 7.3 Git 提交与分支规范

### 提交规范
```
<type>(<scope>): <subject>
```
- 类型：feat / fix / docs / style / refactor / test / chore
- 示例：`feat(algorithm): 添加选品阈值配置接口`

### 分支策略
- `main`：生产分支，只接受 PR 合并，始终可部署
- `develop`：集成分支
- `feature/<模块>-<简述>`：功能分支，完成后提 PR 回 `develop`
- 所有 PR 需附模板：变更说明 / 测试方式 / 是否影响数据库迁移

## 7.4 代码审查要求
- 所有代码合并前需经过至少 1 人审查（如为单人项目，需自我审查 + 测试）
- 审查关注点：逻辑正确性、性能、安全、可维护性
- 审查通过后方可合并到主分支

## 7.5 测试标准

| 测试类型 | 覆盖率要求 | 说明 |
|---------|-----------|------|
| 单元测试 | ≥80% | 核心算法和工具函数必须覆盖 |
| 集成测试 | 关键API全覆盖 | 所有外部API调用需有 Mock 测试 |
| 端到端测试 | 核心流程 | 至少覆盖：选品→匹配→看板展示 |

### 测试框架选型

| 层级 | 框架 |
|------|------|
| 后端单元测试 | pytest + pytest-cov |
| 后端集成测试 | pytest + httpx |
| 前端单元测试 | Jest + React Testing Library |
| 端到端测试 | Playwright（可选） |

## 7.6 性能标准

| 指标 | 阈值 | 告警 |
|------|------|------|
| API p95 响应 | <800ms | >1.5s 告警 |
| DB 慢查询 | <200ms | >500ms 记录 |
| 前端 LCP | <2.5s | >4s 告警 |
| 前端可交互 TTI | <3.5s | — |

## 7.7 CI/CD 要求
- 代码推送后自动运行单元测试 + lint
- 测试通过后自动构建 Docker 镜像（develop 分支）
- 生产环境部署需手动触发（main 分支打 tag）

## 7.8 环境管理

| 环境 | 用途 | 部署方式 | 数据 |
|------|------|---------|------|
| dev | 本地开发 | Docker Compose | mock + 少量真实 |
| staging | 集成测试 | 云服务器（单机） | 真实数据副本 |
| production | 正式使用 | 云服务器（单机） | 真实数据 |

- 三套环境配置通过 `.env` 文件区分，`.env.example` 提交仓库，真实 `.env` 不提交。
- 数据库迁移：使用 Alembic 管理，每次迁移必须包含 `upgrade()` 与 `downgrade()`。

## 7.9 日志规范

### 日志级别使用标准
- DEBUG：开发调试信息
- INFO：关键操作记录（数据同步、API 调用）
- WARNING：非致命异常（外部 API 超时后自动重试）
- ERROR：需要人工介入的错误（数据写入失败）

### 日志格式（结构化 JSON）
```json
{"ts":"2026-07-15T08:00:00Z","level":"INFO","trace_id":"abc123","module":"sync","msg":"Sorftime 同步完成","count":1200}
```
- 每个 HTTP 请求与定时任务注入唯一 `trace_id`，贯穿全链路。
- 禁止打印明文 API Key、token、用户敏感数据（脱敏后仅留后4位）。

## 7.10 安全规范（映射 OWASP Top 10）

- **密钥管理**：所有密钥/密码使用环境变量，禁止硬编码；定期轮换。
- **注入防护**：所有数据库访问使用 ORM 或参数化查询，禁止字符串拼接 SQL。
- **鉴权**：所有 API 必须校验 Bearer Token；管理接口增加 IP 白名单（部署侧 Nginx 配置）。
- **输入校验**：Pydantic 校验所有入参；前端输出做 HTML 转义防 XSS。
- **依赖安全**：使用 `safety` / Dependabot 定期扫描，发现高危依赖 24h 内升级。
- **日志脱敏**：`operation_logs.details` 不得记录明文 Key，写入前过滤。
