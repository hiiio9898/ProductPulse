# 部署与运维方案

## 9.1 部署架构（生产环境）

> v1.0 为单机部署（MVP 取舍）。v1.1 演进路径：PG 主从 + Redis 哨兵 + 应用多副本 + 负载均衡。

```text
┌──────────────┐
│   云服务器   │ (建议配置: 4C/8G/100GB SSD)
│ (Ubuntu 22.04)│
├──────────────┤
│ Docker容器   │
│ ┌────────────┐ │
│ │  Nginx     │ │ :80/:443
│ │ (反向代理) │ │
│ └──────┬─────┘ │
│ ┌──────▼─────┐ │
│ │  FastAPI   │ │ :8000
│ │ (应用服务) │ │
│ └──────┬─────┘ │
│ ┌──────▼─────┐ │
│ │ PostgreSQL │ │ :5432
│ └────────────┘ │
│ ┌────────────┐ │
│ │   Redis    │ │ :6379
│ └────────────┘ │
│ ┌────────────┐ │
│ │  Celery    │ │
│ │  Worker    │ │
│ └────────────┘ │
└──────────────┘
```

## 9.2 部署步骤

```bash
# 1. 服务器初始化
apt update && apt upgrade -y
apt install docker.io docker-compose -y

# 2. 克隆项目
git clone [项目地址] /app

# 3. 配置环境变量（.env文件）
cp .env.example .env
# 编辑 .env 填入 API Keys（禁止提交到 Git）

# 4. 启动服务
cd /app
docker-compose up -d

# 5. 初始化数据库
docker-compose exec app python scripts/init_db.py
```

### `.env.example`（脱敏模板，需提交到仓库）

## 9.3 Docker Compose 配置示例

```yaml
version: '3.8'
services:
  postgres:
    image: postgres:15
    container_name: productpulse_postgres
    restart: always
    environment:
      POSTGRES_DB: productpulse
      POSTGRES_USER: productpulse
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    volumes:
      - pg_data:/var/lib/postgresql/data
    ports:
      - "5432:5432"

  redis:
    image: redis:7
    container_name: productpulse_redis
    restart: always
    ports:
      - "6379:6379"

  app:
    build: ./backend
    container_name: productpulse_app
    restart: always
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/api/v1/health"]
      interval: 30s
      timeout: 5s
      retries: 3
    environment:
      - DATABASE_URL=postgresql://productpulse:${DB_PASSWORD}@postgres:5432/productpulse
      - REDIS_URL=redis://redis:6379
      - SORFTIME_API_KEY=${SORFTIME_API_KEY}
      - ALI1688_APP_KEY=${ALI1688_APP_KEY}
      - GLM_API_KEY=${GLM_API_KEY}
    ports:
      - "8000:8000"
    depends_on:
      - postgres
      - redis

  celery:
    build: ./backend
    container_name: productpulse_celery
    restart: always
    command: celery -A app.celery worker --beat --loglevel=info
    environment:
      - DATABASE_URL=postgresql://productpulse:${DB_PASSWORD}@postgres:5432/productpulse
      - REDIS_URL=redis://redis:6379
    depends_on:
      - app
      - redis

  nginx:
    image: nginx:alpine
    container_name: productpulse_nginx
    restart: always
    volumes:
      - ./nginx/nginx.conf:/etc/nginx/nginx.conf
      - ./frontend/dist:/usr/share/nginx/html
    ports:
      - "80:80"
      - "443:443"
    depends_on:
      - app

volumes:
  pg_data:
```

### Nginx 配置示例（`nginx/nginx.conf`）

```nginx
server {
    listen 80;
    server_name _;
    return 301 https://$host$request_uri;   # 强制跳 HTTPS
}

server {
    listen 443 ssl;
    server_name _;
    ssl_certificate     /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/key.pem;

    # 前端静态
    location / {
        root /usr/share/nginx/html;
        try_files $uri $uri/ /index.html;
    }

    # 后端 API 反代
    location /api/ {
        proxy_pass http://app:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_read_timeout 60s;
    }
}
```

## 9.4 监控告警

| 监控项 | 工具 | 告警阈值 |
|--------|------|---------|
| 服务存活 | Health Check (/api/v1/health) | 连续 3 次失败 → 告警 |
| API响应时间 | 应用日志 | >5 秒 → 告警 |
| 定时任务执行 | Celery 监控 (Flower) | 任务失败 → 告警 |
| 磁盘使用率 | 云监控 | >80% → 告警 |
| API调用失败率 | 应用日志 | >10% → 告警 |

## 9.5 日志管理

- 应用日志输出到 stdout，Docker 通过 `docker logs` 或日志驱动收集。
- **日志轮转**：配置 docker `json-file` 驱动限制 `max-size=50m, max-file=5`，避免磁盘撑满。
- 日志保留 7 天热数据，超期归档至冷存储。

```yaml
# docker-compose.yml 顶层日志配置
logging:
  driver: json-file
  options:
    max-size: "50m"
    max-file: "5"
```

## 9.6 备份策略

| 备份内容 | 频率 | 保留周期 |
|---------|------|---------|
| PostgreSQL 数据库 | 每日凌晨 2:00 | 30 天 |
| 系统配置文件 | 每次变更时手动 | 永久 |
| 日志文件 | 每日归档 | 7 天 |

> **恢复演练**：每月执行一次从备份恢复到测试环境的演练，验证备份可用性。

### 自动备份脚本示例

```bash
# 加入 crontab: 0 2 * * * /app/scripts/backup.sh
docker-compose exec -T postgres pg_dump -U productpulse productpulse | gzip > /backups/pp_$(date +%F).sql.gz
find /backups -name "pp_*.sql.gz" -mtime +30 -delete   # 清理 30 天前
```

## 9.7 故障恢复

| 场景 | 恢复方式 | 预估时间 |
|------|---------|---------|
| 应用服务崩溃 | Docker 自动重启 (restart: always) | <1 分钟 |
| 数据库损坏 | 从最近备份恢复 | <30 分钟 |
| 外部 API 不可用 | 重试机制 + 降级提示 | 实时 |

## 9.8 版本回滚

- 每次发布使用唯一镜像 tag（`productpulse:v1.0.3`，禁止 `latest` 上生产）。
- 回滚步骤：`docker-compose down` → 修改镜像 tag 至上一稳定版本 → `docker-compose up -d`。
- 数据库迁移必须提供 `downgrade()` 脚本，回滚时先回滚代码再回滚迁移。
